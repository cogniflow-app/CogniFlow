# Phase 02 product UX audit

Date: 2026-07-21  
Scope: the completed Phase 02 product only; no scheduling, mastery, review, database, API, authorization, or environment changes.

## Method and evidence

The audit combined a source and route inventory with a sanitized local Supabase account, the running Next.js application, Playwright interaction checks, and direct inspection of captured browser pixels. The built-in browser bridge was unavailable in this environment, so the same local application was exercised through the repository's isolated Chromium runner. Screenshots remain uncommitted test artifacts.

The common problems are systemic rather than page-specific:

- Product pages have several independently evolved shells instead of one reliable page, header, toolbar, readable-copy, and action system.
- Shared SVG icons had no explicit size. Once protected from flex shrinking, the browser's oversized default SVG dimensions overflowed the action and compressed its label, making the purple button appear to end before its text. Button and LinkButton now size direct icons and keep icon/label geometry stable together.
- The internal note/generated-card model leaks into ordinary authoring copy. It is accurate for storage but makes “add a flashcard” feel like a database operation.
- The generic rich-text editor is repeated for compact answers and optional feedback. Multiple-choice and ordering therefore spend more space on toolbars than decisions.
- Tags, source, aliases, language, feedback, matching settings, templates, and CSS compete with required fields instead of appearing progressively.
- Desktop layouts are generally polished at the largest breakpoint, but intermediate widths depend too much on flex shrinking. Long titles, actions, and filter groups need earlier wrapping or substitution.
- Legal/auth pages are accessible, but their readable-width and action alignment rules are not explicitly shared with product pages.

## Acceptance evidence after implementation

All baseline rows above were rechecked against the following evidence sets:

- **E1 — component behavior:** focused content tests cover every card type, friendly validation, progressive disclosure, choice semantics/reordering, cloze selection, save-and-add-another, true/false selection, visual-region synchronization, media states, type switching, exact saved-draft navigation protection, and Library behavior. The UI package adds variant, loading, disabled, icon, long-label, and LinkButton geometry assertions.
- **E2 — authenticated workflow:** the Chromium content-authoring acceptance creates a typed-answer deck, edits advanced metadata, saves, opens card previews, publishes, copies/opens its link, checks the public and embed players, unpublishes, and cleans up through real local APIs.
- **E3 — layout pixels:** the Phase 02 layout test captures empty/populated Library, deck creation, Overview, card browser, History, Settings, Published, every card composer, image occlusion before/after upload, public card front/back/mobile, dark serious mode, and reduced motion. Screenshots are sanitized uncommitted artifacts and were opened for pixel inspection.
- **E4 — responsive geometry:** computed assertions cover 1920×1080, 1536×1024, 1440×900, 1366×768, 1366×700, 1280×800, 1024×768, 768×1024, 430×932, 390×844, 360×800, and 320×568, plus 125%/150% zoom and 200% text. They assert no document overflow, viewport-contained controls, separated filters, matched flashcard faces, non-mirrored answer content, toolbar gaps, and button-child bounds.
- **E5 — accessibility:** the a11y suite covers public, authentication, onboarding, policy, workspace, design-system, drawer focus trap/return, appearance, keyboard deck/card authoring, unsaved-navigation recovery, both themes, serious mode, and reduced motion with no serious or critical axe violations.
- **E6 — implementation boundaries:** the diff contains no migrations, database/RLS/API/environment changes, authentication behavior changes, Phase 03 scheduling/review work, fake statistics, or committed browser artifacts. Production-only design-system access resolves to not-found.

## Acceptance evidence checklist

The completed audit will link each correction to:

- component/unit assertions for terminology, disclosure, selection, and mutation behavior;
- Playwright computed-layout assertions for gutters, overflow, button contents, menus, dialogs, and card bounds;
- sanitized screenshots for every major route/card type at desktop and mobile;
- explicit checks at the required eleven viewports, 125% and 150% zoom, 200% text, long content, both themes, serious mode, reduced motion, and keyboard-only operation;
- the complete local validation command matrix and hosted Preview smoke result.

## Phase 03 study UX extension

Phase 03 preserves the compact workspace and adds one first-class Study destination rather than a
second dashboard. Study uses concise deck rows and queue controls; Settings hides advanced
scheduler details by default; Statistics uses real tables and restrained summaries instead of a
metric wall. Ordinary copy says Card, Related cards, requested retention, and due time rather than
exposing internal card-generation or raw FSRS structures.

The review surface keeps answer content out of DOM before reveal, makes four ratings visible only
after reveal, uses text plus color, supports `1`–`4`, never grades on swipe alone, and keeps the card
when commit fails. It includes explicit empty, paused, conflict, completed, content-change, and undo
states.

Dedicated Chromium layout acceptance covers empty/populated Study, deck entry, custom study,
collapsed and expanded scheduling settings, prompt and answer faces, all ratings, undo, statistics,
per-card history, semantic content decisions, serious mode, reduced motion, 320 px mobile, and 200%
text without horizontal overflow. Two visual projects pass. Actual repository-produced browser
pixels were opened and inspected for populated Study, scheduling progressive disclosure, and mobile
statistics at 200% text; the in-app browser runtime was unavailable, so direct visual QA used the
local image viewer. The full browser suite passes 29 scenarios with 19 intentional cross-project
skips, and the accessibility suite passes 28/28 checks. No screenshot or browser artifact is
tracked.
